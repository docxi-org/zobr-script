# ZS (Zobr Script) — Language Specification v0.1

## 1. Overview

ZS is a cognitive scripting language for describing reasoning processes. It provides formal constructs for structuring thought — not as rigid instructions, but as a composable vocabulary of cognitive operations with variables, control flow, and result formatting.

ZS scripts can be:
- **Written by a human** as a task for an LLM agent
- **Generated by an agent** as a reusable artifact from a reasoning session
- **Stored and shared** between agents via KB or file system

File extension: `.zobr`

### Design principles

1. **Operations have fixed semantics** — `contrast` always means "find an opposing position", not whatever the model decides
2. **Variables carry context** — results flow between operations as named semantic objects
3. **Structure guides, not constrains** — the agent fills operations with content freely
4. **Readable by both humans and LLMs** — minimal syntax, natural language inside operations

---

## 2. Script structure

```
task: "<question or problem statement>"
input: <optional named inputs>

<body: sequence of operations>

result = conclude {
  <output field definitions>
}
```

A script starts with a `task` declaration, optionally declares `input` parameters, contains a body of cognitive operations, and ends with a `conclude` block that defines the output format.

## 3. Built-in operations

### 3.1 Discovery operations

#### `survey(topic, count?: N)`
Explore a topic and identify key elements (positions, factors, perspectives).

```
positions = survey("main positions on consciousness", count: 3)
// Returns: list of identified items with brief descriptions
```

#### `ground(claim, extract?: [fields])`
Connect a claim or topic to concrete evidence, facts, or experience.

```
evidence = ground(news_article, extract: [facts, quotes, dates])
// Returns: structured evidence base
```

### 3.2 Argumentative operations

#### `assert(thesis, based_on?: source)`
State a position with reasoning.

```
claim = assert("consciousness is computational", based_on: evidence)
// Returns: thesis with supporting argument
```

#### `doubt(target)`
Problematize a claim — find weaknesses, hidden assumptions, edge cases.

```
weakness = doubt(claim)
// Returns: identified problems with the target claim
```

#### `contrast(target, with?: alternative)`
Find or construct an opposing position or counterexample.

```
counter = contrast(claim)
counter = contrast(claim, with: "phenomenological perspective")
// Returns: opposing position with its argument
```

#### `analogy(target, from?: domain)`
Transfer understanding from another domain.

```
insight = analogy(problem, from: "evolutionary biology")
// Returns: analogical mapping with source and target
```

### 3.3 Synthesis operations

#### `synthesize(sources, method?: approach)`
Combine multiple positions or findings into a higher-level understanding.

```
unified = synthesize([pos_a, pos_b, pos_c], method: "find common ground")
// Returns: synthesized position
```

#### `reframe(target, lens?: perspective)`
Reformulate a problem or position in different terms.

```
new_view = reframe(problem, lens: "cui bono")
// Returns: reformulated problem statement
```

### 3.4 Meta-operations

#### `assess()`
Reflective pause — evaluate the current state of reasoning.

```
state = assess()
// Returns: { status: open|converging|stuck, tension: string, missing: string }
```

#### `pivot(reason)`
Explicitly change reasoning strategy.

```
pivot("linear analysis insufficient, switching to dialectical approach")
// Side effect: signals strategy change, no return value
```

#### `scope(direction, focus?)`
Narrow or widen the focus of inquiry.

```
scope(narrow, "what specifically distinguishes understanding from pattern-matching")
scope(wide, "how does this connect to embodied cognition")
```

### 3.5 Output operation

#### `conclude { fields }`
Define the structure and format of the final result.

```
result = conclude {
  thesis: string           // 1-2 sentence answer
  confidence: low | medium | high
  key_evidence: list       // supporting points
  open_questions: list     // what remains unresolved
  next_steps: list         // suggested follow-up
}
```

Field types:
- `string` — free text
- `list` — list of items
- `low | medium | high` — enumeration
- `bool` — true/false
- `structured` — nested object

## 4. Variables

Variables are assigned with `=` and hold semantic objects (not just strings).

```
positions = survey("...")          // list
claim = assert("...")              // thesis object
weakness = doubt(claim)            // critique object
result = conclude { ... }          // structured output
```

### References

- `@last` — result of the previous operation
- `@N` — result of the N-th operation (1-indexed)
- Named variables — preferred for clarity

```
a = assert("...")
b = doubt(@last)          // same as doubt(a)
c = contrast(@1)          // same as contrast(a)
```

## 5. Control flow

### 5.1 Iteration

```
for item in collection {
  <operations using item>
}
```

Example:
```
positions = survey("views on free will", count: 3)
arguments = for p in positions {
  pro = ground(p, extract: [strongest_argument])
  contra = contrast(p)
  yield { position: p, pro, contra }
}
```

`yield` inside `for` collects results into a list.

The iterable can be a variable, operation call, or an **array literal**:
```
for lang in ["Python", "Rust", "Haskell"] {
  strengths = survey(lang, count: 3)
  yield { language: lang, strengths }
}
```

### 5.2 Conditionals

```
if <condition> {
  <operations>
} else {
  <operations>
}
```

Conditions reference assessment results or variable properties:

```
state = assess()
if state.status == stuck {
  pivot("trying different approach")
  reframe(@task)
} else {
  synthesize(collected)
}
```

### 5.3 Loops

```
loop until <condition> {
  <operations>
}

loop N times {
  <operations>
}
```

Example:
```
thesis = assert("initial position")
loop 3 times {
  challenge = doubt(thesis)
  if challenge.holds {
    thesis = reframe(thesis, considering: challenge)
  }
}
// thesis has been stress-tested 3 times
```

## 6. User-defined functions

Functions are defined with `define`. The body contains a `prompt` (natural language instruction for the LLM) and declares inputs/outputs.

```
define steelman(position) {
  prompt: "Find the strongest possible version of this position.
    Strengthen the arguments, remove weak points, add missing
    evidence. Be charitable and rigorous."
  input: position
  output: strengthened_position
}

define socratic_probe(thesis, depth: 3) {
  prompt: "Ask a question that exposes a hidden assumption
    in the thesis. Then reformulate the thesis without
    that assumption."
  loop depth times {
    question = ask(thesis)
    assumption = extract(question, type: hidden_assumption)
    thesis = reframe(thesis, without: assumption)
  }
  output: { refined_thesis, exposed_assumptions: list }
}
```

Usage:
```
claim = assert("AI will achieve AGI by 2030")
strong = steelman(claim)
probed = socratic_probe(strong, depth: 2)
```

## 7. Libraries and imports

```
import <library_path>
```

Libraries are collections of user-defined functions stored as `.zobr` files.

```
import philosophy/dialectics
import analysis/news

positions = survey("...")
for p in positions {
  p = dialectics.thesis_antithesis_synthesis(p)
}
report = news.structured_analysis(article)
```

Standard library paths:
- `kb://scripts/<name>` — stored in Zobr KB
- `./local_lib.zobr` — local file
- `stdlib/<name>` — built-in library (future)

## 9. Complete example

```
// news-analysis.zobr — Structured political news analysis
// Generated from Evald + Dmitry dialogue session

task: "Analyze a political news article for hidden interests"
input: news_text

// Extract factual base
source = ground(news_text, extract: [facts, quotes, dates, actors])

// Identify stakeholders
stakeholders = survey("who benefits or loses", from: source, count: 5)

// Analyze each stakeholder
analysis = for s in stakeholders {
  motive = assert(s.interest, based_on: source)
  hidden = doubt(motive)
  credibility = assess()
  yield { actor: s, motive, hidden_agenda: hidden, credibility }
}

// Find the narrative
official = ground(news_text, extract: [stated_reasons])
actual = synthesize(analysis, method: "compare stated vs probable motives")
gap = contrast(official, with: actual)

// Reframe
deeper = reframe(news_text, lens: "cui bono")

result = conclude {
  summary: string
  what_happened: string
  official_narrative: string
  probable_reality: string
  who_benefits: list
  who_loses: list
  what_is_hidden: list
  confidence: low | medium | high
  watch_for: list        // what to monitor next
}
```

## 10. Execution model (v0.1)

In v0.1, ZS scripts are executed by an LLM as interpreter:

1. The script is included in the system prompt or user message
2. The LLM reads the script and executes it step by step
3. At each operation, the LLM generates content appropriate to that operation's semantics
4. Variables are tracked by the LLM in its context
5. The final `conclude` block structures the output

The LLM is expected to:
- Show which operation it is executing (e.g., `[survey]`, `[doubt]`)
- Display variable assignments
- Follow control flow (loops, conditionals)
- Produce output matching the `conclude` format

Future versions may introduce:
- External runtime (ThinkScript VM) for managing state
- MCP integration for operation dispatch
- Hybrid execution (runtime + LLM)

## Appendix A: Operation summary

| Operation | Category | Input | Output | Semantics |
|-----------|----------|-------|--------|-----------|
| survey | Discovery | topic, count? | list | Explore and identify key elements |
| ground | Discovery | claim, extract? | evidence | Connect to concrete facts |
| assert | Argument | thesis, based_on? | claim | State a position with reasoning |
| doubt | Argument | target | critique | Find weaknesses and assumptions |
| contrast | Argument | target, with? | counter | Construct opposing position |
| analogy | Argument | target, from? | mapping | Transfer from another domain |
| synthesize | Synthesis | sources, method? | unified | Combine into higher understanding |
| reframe | Synthesis | target, lens? | new_view | Reformulate in different terms |
| assess | Meta | — | state | Evaluate current reasoning state |
| pivot | Meta | reason | — | Change reasoning strategy |
| scope | Meta | direction, focus? | — | Narrow or widen inquiry focus |
| conclude | Output | fields | result | Structure final output |

## Appendix B: Reserved words

```
task, input, result, conclude, define, import,
for, in, if, else, loop, until, times, yield,
survey, ground, assert, doubt, contrast, analogy,
synthesize, reframe, assess, pivot, scope,
string, list, bool, low, medium, high, structured,
true, false, null
```

Reserved words cannot be used as **variable names**, **function names**, or **parameter names**. However, they **can** be used as:

- **Object property keys**: `{ task: "analysis", result: data }`
- **Conclude field names**: `conclude { input: source, result: findings }`
- **Named argument labels**: `ground(data, extract: [facts])`

This distinction allows natural naming in output structures while preventing ambiguity in declarations and control flow.
