# Классы доверия

Каждое событие в [трассировке](trace) помечено **классом доверия**. Он отвечает на один вопрос: *откуда содержание этого результата?*

| Класс | Цвет | Значение |
|---|---|---|
| `asserted` | янтарный | Произведено агентом (рассуждение модели или данные, переданные агентом). Правдоподобно, независимо не проверено. |
| `verified` | зелёный | Произведено или проверено детерминированным кодом — sandbox, оценка checkpoint, валидация формы. |
| `authority` | синий | Получено от пользователя через human-in-the-loop (`ask_user`). Ground truth по определению. |
| `n/a` | серый | Инфраструктурное событие (start, переход статуса). Не вклад в результат. |

## Как назначается доверие

Доверие определяется по **происхождению содержания**, а не просто по тому, кто выполнил операцию:

- Когнитивная операция (`survey`, `doubt`, `synthesize`) — содержание от модели → `asserted`
- `report` — сервер записывает то, что прислал агент, но содержание произведено агентом → `asserted`
- `act` — результат действия, пересказанный агентом → `asserted`
- Вызов [серверной функции](server-module) / `@sandbox` — детерминированное вычисление → `verified`
- [Контрольная точка](checkpoints) — сервер оценивает данные и возвращает директиву → `verified`
- `retrieve` с provenance (агент получил данные из внешнего инструмента и указал источник) → `verified`; без provenance → `asserted`
- `conclude` валидируется по форме → `verified`
- Ответ `ask_user` от пользователя → `authority`
- `start` или переход статуса → `n/a` (инфраструктура, исключается из покрытия)

> **Заметка:** результат `asserted` не означает ошибочный — он означает *непроверенный*. Трассировка делает это различие видимым, чтобы потребители могли решить, сколько верификации им нужно.

## Доверие и трассировка

Метрика [покрытия](coverage) агрегирует доверие по всем событиям: какая доля работы верифицирована vs утверждена. Инфраструктурные события (`n/a`) исключаются из подсчёта. Трассировка из одних `asserted` шагов — гипотеза; подкреплённая `verified` и `authority` — аудированное утверждение.

Бейджи доверия отображаются на каждой строке хронологии событий в детальном просмотре [трассировки](trace). [Контрольные точки](checkpoints) всегда `verified`, потому что сервер оценивает и отвечает на них.
