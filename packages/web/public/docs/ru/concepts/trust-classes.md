# Классы доверия

Каждое событие в [трассировке](trace) помечено **классом доверия**. Он отвечает на один вопрос: *как был получен этот результат?*

| Класс | Цвет | Значение |
|---|---|---|
| `asserted` | янтарный | Модель предложила на основе рассуждения. Правдоподобно, независимо не проверено. |
| `verified` | зелёный | Проверено по объявленной форме, правилу или детерминированному вычислению. |
| `authority` | синий | Получено от доверенной границы — [серверная функция](server-module), внешний I/O, хост. |
| `error` | красный | Шаг завершился ошибкой. Содержит `kind` и `message`. |

## Кто назначает доверие

Доверие назначает **реализатор** — компонент, который фактически произвёл результат:

- Когнитивная операция (`survey`, `doubt`, `synthesize`) реализуется моделью → `asserted`
- Вызов [серверной функции](server-module) реализуется сервером → `authority`
- [Контрольная точка](checkpoints) проверяется сервером → `verified`
- `conclude` валидируется по форме → `verified`
- Событие `start` записывается сервером → `verified`

> **Заметка:** результат `asserted` не означает ошибочный — он означает *непроверенный*. Трассировка делает это различие видимым, чтобы потребители могли решить, сколько верификации им нужно.

## Доверие и трассировка

Метрика [покрытия](coverage) агрегирует доверие по всем событиям: какая доля работы верифицирована vs утверждена. Трассировка из одних `asserted` шагов — гипотеза; подкреплённая `verified` и `authority` — аудированное утверждение.
