# Hot/cold жизненный цикл

Сервер управляет вызовами в двух состояниях:

## Hot (в памяти)

Горячий вызов хранит полное состояние в памяти: объект Instance, процесс worker (для скриптов с [серверным модулем](server-module)), события трассировки, хранилище handle'ов. Готов принять следующий MCP-вызов немедленно.

## Cold (снимок в SQLite)

При вытеснении сервер сериализует полное состояние — включая свойства экземпляра класса worker, структуры данных Set/Map — в JSON-снимок в SQLite. Worker уничтожается, память освобождается.

## Что вызывает вытеснение

| Триггер | Когда |
|---|---|
| **TTL** | Вызов не был активен в течение `invocationTtlMs` (по умолчанию: 1 час) |
| **TTL ожидания** | Приостановленный вызов ждёт слишком долго (по умолчанию: 24 часа) → `expired` |
| **LRU** | Превышен `maxActiveInvocations` → вытесняется самый старый по `lastActivityAt` |
| **Sweep** | Проверка TTL происходит при `zs_start` и `zs_register` |

## Восстановление

Вытесненный вызов можно восстановить:

```
zs_resume({ invocation_id: "inv_..." }) → полное состояние восстановлено
```

Сервер загружает снимок, пересоздаёт worker, восстанавливает состояние экземпляра класса (включая Set и Map с тегами типов), и вызов снова горячий — готов продолжить с того места, где остановился.

При попытке обратиться к вытесненному вызову без восстановления сервер возвращает:

```json
{ "error": { "kind": "evicted", "message": "call zs_resume to restore" } }
```

## Конфигурация

Задаётся в `.env`:

| Переменная | По умолчанию | Значение |
|---|---|---|
| `ZS_INVOCATION_TTL` | 3600 (1ч) | Секунды до вытеснения при простое |
| `ZS_AWAITING_TTL` | 86400 (24ч) | Секунды до перехода suspended → expired |
| `ZS_MAX_ACTIVE_INVOCATIONS` | 100 | Лимит LRU |
